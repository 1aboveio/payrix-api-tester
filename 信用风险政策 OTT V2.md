## 1. 政策声明

### 1.1 引言

审慎的风险管理是任何金融或支付机构确保持续财务可行性和运营弹性的基础。本政策明确规定了管理信用风险的标准、流程和程序。它建立了识别、评估和缓释已识别信用和某些财务风险敞口的框架。其他关键的商户风险类型，包括欺诈风险、数据安全事件风险（数据泄露）和受理风险，在单独的专门政策框架中予以处理和管辖。

在本政策背景下，信用风险被定义为商户蒙受财务损失的潜在可能性。这种风险主要源于商户无法或未能履行其财务义务。信用风险的具体驱动因素包括商户破产或资不抵债、因发卡银行或持卡人发起的拒付而导致商户不结算所产生的敞口。此类敞口可能导致针对商户组合持有的财务抵押品不足，从而可能阻碍支付服务商履行其对卡组织的结算义务的能力。

本政策阐明了风险职能部门用于缓释商户信用风险敞口并确保履行对合作伙伴和卡组织义务的方法和高层次流程。它建立了在初始尽职调查（入网）期间以及通过持续组合监控评估商户财务可行性的框架。本政策进一步阐述了实施财务缓释措施的要求，如支付服务商（或PSP）持有的抵押品或准备金，以应对潜在的结算或信用风险事件。它还涵盖定期审查流程和其他相关风险控制措施。

本文档不包含政策执行所需的详细操作程序。

在本政策中，客户、顾客和商户等术语可互换使用。

### 1.2 信用风险管理职责

商户信用风险治理和管理的主要职责分配给风险职能部门。在初始承保阶段（入网）期间，风险职能部门被要求评估潜在敞口，并根据商户信用风险评分模型的输出确定适当的财务缓释措施（如抵押品），如本政策后续章节进一步阐述。在商户整个生命周期中，风险职能部门持续负责持续风险监控、定期审查以及必要调整或建立更新的财务安全要求。

### 1.3 定义和公式

- **总敞口（Gross Exposure）：** 计算为（年交易总额TPV美元除以365，乘以平均履约期）、总拒付金额（美元）和总退款/贷记金额（美元）之和。

- **敞口覆盖率（EC）：** 通过抵押品除以总敞口确定。

- **预期敞口（EE）：** 这是总敞口与风险因素的乘积，根据商户信用风险评级（1至5级）应用不同的百分比。
  
  - 评级1：EE设为零，意味着预计无风险。
  - 评级2：EE计算为总敞口的5%。
  - 评级3：EE为总敞口的15%。
  - 评级4：EE为总敞口的30%。
  - 评级5：EE为总敞口的50%。

- **净预期敞口（NEE）：** 通过从预期敞口（EE）中减去抵押品计算。

- **风险/回报比率（回收期）：** 以月为单位表示，计算为NEE（如果大于0，代表无担保敞口）与月预期净收入的比率。

### 1.4 禁止和受限行业

本机构（本文中称为支付服务商或PSP）致力于通过维护明确分类的行业清单来实施主动风险管理，这些行业被确定为呈现较高的固有风险特征。该分类采用稳健的方法论制定，纳入多样化的数据输入，以促进对潜在风险的全面理解和有效缓释。

将行业归类为高风险的判定标准包括但不限于：

- **商户类别代码（MCC）：** 列入卡组织关于高品牌风险或加强监控的通告。
- **监管要求：** 根据相关收单司法管辖区或适用的反洗钱（AML）框架的法规被认定为高风险或禁止的行业。
- **固有业务模式属性：** 具有与信用敞口、声誉影响或监管合规复杂性相关的基本风险因素的行业。
- **历史组合表现：** 实证数据分析显示持续存在上述类别未涵盖的高风险指标模式，符合本机构定义的风险偏好。

下表总结了行业风险管理方法的关键方面：

| 高风险行业判定标准   | 描述         | 监控频率 | 益处     |
| ----------- | ---------- | ---- | ------ |
| 商户类别代码（MCC） | 高风险代码      | 每年   | 加强内部控制 |
| 监管要求        | 当地监管的高风险行业 | 持续   | 监管合规   |
| 固有业务模式属性    | 信用、声誉或监管风险 | 每年   | 适应变化   |
| 历史组合表现      | 有高风险历史的行业  | 持续   | 财务稳定   |

为确保分类保持时效性和准确性，本机构实施持续监控和定期审查流程：

- **年度审查：** 分类至少每年接受一次全面审查，以与当前风险状况和不断演变的监管环境保持一致。
- **持续监控：** 保持持续监控流程，以促进主动识别和及时评估新兴风险因素和行业发展。

对禁止和受限行业分类的勤勉管理显著增强了风险控制措施，巩固了监管合规，并支撑了财务稳定。它进一步有助于维护本机构的声誉地位，并有助于维护安全的交易处理环境。

关于归类为禁止或受限行业的详尽清单，请参阅《商户入网政策》中标题为"禁止和受限商户类别"的专门章节。本文档提供了OTT认定为呈现高风险特征的具体行业的详细说明。

## 2. 信用风险承保概述

### 2.1 触发事件

OTT采用基于风险的方法来处理信用风险承保流程，将在以下情况下触发并对商户账户进行：

1. **新商户入网：** 当新商户加入支付收单服务时。
2. **现有客户激活：** 当当前客户开始使用支付收单服务时。
3. **定期审查：** 定期发生或在满足某些标准时发生。

在此过程中，商户必须提供特定信息，尽管并非所有财务细节都是强制性的。

### 2.2 进行评估

在提交所需文件后，进行全面的财务风险评估。该评估考虑各种要素，包括：

- **商户背景：** 分析商户的历史和资质。
- **网站/应用程序审查：** 检查商户的在线存在。
- **财务审查：** 详细查看商户的财务报表。
- **处理历史：** 评估商户过去的行为和模式。

然后根据这些标准应用评分系统对每个商户进行评级。

### 2.3 确定风险等级

评估完成后，为每个商户分配商户信用风险评级。该评级与风险因素一起用于估计潜在风险敞口。如果商户未能提供必要文件，他们可能会获得更高的风险评级和更大的风险因素。

### 2.4 信用风险缓释

评估完成后，风险团队概述商定的抵押品条款。这些可能包括：

- **滚动准备金：** 轮换基础上预留的资金。
- **固定准备金：** 预留的预定金额资金。
- **结算延迟：** 结算流程的延期。
- **结算频率：** 结算支付的规律性。
- **银行担保：** 金融机构提供的担保。
- **母公司担保：** 来自母公司的支持。
- **个人担保：** 关键人员的个人担保。
- **月处理限额：** 月交易量的上限。

这些措施旨在缓释已识别的信用风险并确保交易的财务稳定性。

## 3. 信用风险承保流程

### 3.1 商户背景

**基本信息核实**

OTT必须核实商户的基本信息，包括企业注册类型、投资资本、员工规模、股东详情、历史业务表现以及近期和长期战略计划。通过查阅公共记录（如政府数据库）或其他可信信息提供商进行核实。

**业务运营洞察**

对商户运营活动的彻底审查对于理解业务持续运营的逻辑至关重要。这包括对盈利模式的深入分析，涉及提供的产品或服务、目标客户群、竞争分析和供应链管理。

**物流和交付评估**

必须关注物流和交付方面，因为它们是商户产品的关键组成部分。需要详细的商户物流绩效和历史记录数据进行全面评估。订单履行和交付的及时性显著影响风险敞口评估，需要精确的计算方法。

### 3.2 网站/应用程序审查

详情请参阅《商户入网政策》中的"网站/应用程序审查"章节。

### 3.3 财务审查

财务报表在评估商户业务质量及其可持续性方面至关重要。财务审查使风险团队能够核实业务模式并深入了解业务的运营和财务驱动因素。评分卡工具用于评估商户的财务能力，重点关注资产负债表和损益表的关键财务数据。

虽然内部评级模型不明确加权信息质量，但最好获取高质量数据。可接受的文件包括来自认可会计师事务所的经审计年度报告、未经审计的年度报告和内部编制的财务报表。如果此类文件不可用，内部评级应默认为新商户的3级，以确保足够的安全性。对于已建立商户的持续审查，同样适用默认评级，假设对商户的处理活动和业务运营有一定程度的了解。

### 3.4 处理历史

审查商户的处理历史对于评估财务风险和建立内部风险评级至关重要。这包括总支付量（TPV）、退款率、争议率、拒付率和使用的支付方式组合等指标。这些指标反映了商户产品和服务的质量以及特定时期内的典型商业行为。

为确认处理历史的真实性，应满足特定标准，如收单机构的官方文件，包括标识和参考编号，最好以月度摘要形式呈现。标准审查期为三个月，对于被认定为高风险的商户延长至六个月。对于没有收单历史的商户，应考虑行业平均值。

## 4. 信用审批

### 4.1 信用决策流程

OTT的信用风险评估是定量的，利用评分卡评估商户的信用状况。此流程帮助承保人识别每个商户的风险类别，并建议适当的保障水平。信用评分与信用风险评级相关联，决定预期敞口率：

| 评级  | 预期敞口（覆盖率） |
| --- | --------- |
| 1   | 0         |
| 2   | 5%        |
| 3   | 15%       |
| 4   | 30%       |
| 5   | 50%       |

总敞口包括潜在的订单履行延迟和运营风险。作为评分卡一部分的承保人敞口计算表用于估计商户的整体风险敞口，推荐准备金水平，并确保总敞口覆盖率符合建议的最低要求。虽然敞口覆盖率水平作为参考，但承保人有权调整，须经管理层批准。

### 4.2 风险缓释措施

在风险评估阶段之后，风险职能部门负责正式记录确定的财务缓释要求和相关条款。这些条款必须准确记录，并符合定义的审批和升级协议。可应用的潜在财务保障套件包括但不限于：

- 滚动准备金
- 固定准备金
- 结算延迟
- 结算频率调整
- 银行担保
- 母公司担保
- 个人担保
- 月处理限额

在检测到商户绩效恶化或风险敞口增加的情况下，规定以下升级风险缓释措施：强制对相关交易进行3D Secure认证、修改结算时间表或频率、暂时暂停结算支付、停止交易处理，以及最终根据合同条款终止账户。任何撤销或修改这些缓释措施的行动，如恢复结算或交易处理，都需要严格遵守相同的既定审批和升级协议，并必须全面记录。

升级矩阵规定了各种缓释措施所需的审批级别：

- 实施3DS在风险评级1至4级执行。
- 调整结算频率在风险评级1至3级处理。
- 暂停结算在评级4级管理。
- 暂停交易和终止行动在评级5级执行。

### 4.3 升级协议

**总敞口阈值**

既定流程包含与评估阶段识别的特定风险指标相关的定义审批阈值。超过预定义风险承受能力水平的案例需要升级审查，并升级至指定的二级权限（如风险经理）。二级权限随后负责确定是否基于识别风险的复杂性或严重性需要进一步升级至三级权限（如风险委员会）。

- 总敞口低于10万美元：需要信用风险分析师审批。
- 总敞口低于50万美元或敞口覆盖率（EC）为0%：需要风险经理审批。
- 总敞口50万美元或以上或敞口覆盖率（EC）为0%：需要风险委员会审批。

**拒付率历史**

基于拒付率和支付量历史的商户入网升级流程确保所有商户遵守安全和合规的支付条款以及风险管理策略，有效控制和缓释潜在风险。

- 拒付率低于0.75%：无需升级。
- 拒付率低于0.90%：需要风险经理审批。
- 拒付率0.90%或以上：需要风险委员会审批。
- 无历史或少于3个月且月销售额低于50,000：需要信用风险分析师审批。
- 无历史或少于3个月且月销售额超过50,000：需要风险经理审批。

## 5. 定期审查

### 5.1 商户敞口和评分的定期重新评估

规定的时间表概述了对已批准商户进行定期风险审查的强制性最低频率。每个定义的风险层级需要从适当的授权机构获得认可，具体审批要求与分配的风险评级相称。每次定期评估后，必须编制正式审查报告，记录关键观察、风险发现、结论和建议。

以下是针对不同商户风险评级的审查流程摘要：

- 评级1的商户每半年接受部分审查。
- 评级2的商户每年接受部分审查。
- 评级3、4和5的商户都需要每年接受全面审查。

为这些定期重新评估的目的，必须收集相关数据，收集信息的范围与商户分配的风险分类相称。需要获取的关键数据点包括但不限于：

- 反映近期财务表现的当前管理账户。
- 更新的业务概况和运营文件。
- 最新的可用财务报表（经审计或未经审计）。
- 全面审查商户的网站存在和运营政策，包括条款和条件、隐私政策、配送政策和退款政策。
- 历史拒付率和信用表现历史分析。

这种结构化的重新评估流程促进对每个商户动态风险状况的持续监控，并确保及时实施必要的风险缓释策略调整。

### 5.2 定期交易限额审查

对商户处理上限的月度监督是强制性的。审查财务是信用分析师的责任，除非在指定监控期内已进行近期评估。如果提供的数据不足或不符合要求标准，可实施临时措施。这些可能包括暂停交易、暂停资金或终止关系。

为管理交易量的显著增加，制定了具体协议：

- 对于处理的最初两个月，如果请求增加交易限额，则进行升级审查。
- 对于六个月内首次增加，作为控制措施的一部分触发网站审查。
- 对于六个月内任何后续增加，同时启动网站审查和增强尽职调查（EDD）流程。

这种结构化方法确保对交易量的任何重大变化进行仔细管理，并对风险进行适当评估和缓释。

### 5.3 基于规则的审查

实施信用风险监控规则的目的有两个：跟踪商户绩效趋势和确定每个商户呈现的精确敞口水平。此流程还有助于查明预测风险因素与实际绩效指标之间的任何不一致。

在进行交易限额审查或遵循审查时间表之前，监控报告在发现任何系统性绩效问题方面发挥着关键作用。这使管理层能够清楚了解手头的敞口水平。

日常监控涵盖一系列关键绩效指标，包括：

- 授权成功率
- 退款发生率
- 拒付普遍程度
- 拒付原因代码分布
- 欺诈活动发生
- 交易量突然增加
- 交易量波动
- 处理支付总额
- 高风险商户类别代码比例

这些指标在不同滚动时间段内观察，如最近24小时、最近7天或最近30天，以提供绩效的动态视图。至关重要的是，任何绩效问题都应通过审查及时解决，如有必要，应修订风险评估参数或交易限额。任何修订的评估必须按照组织预定义的审批层级和风险承受能力获得批准。

总体而言，日常审查流程充当保障，确保及时识别和管理任何异常高的敞口或意外的拒付激增。

---

## 6. 政策例外

如标准政策的偏离对实现OTT目标至关重要，必须获得风险委员会的批准。风险管理团队有责任细致记录任何此类例外。此记录应包括例外的详细说明、其背后的逻辑以及预期结果。这些信息必须在随后的风险委员会会议上提交和讨论。

此外，任何例外的认可必须通过电子邮件往来正式确认，确保决策过程清晰可追溯的记录。
