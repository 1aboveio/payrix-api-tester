# 2026-02-19 — CI/CD Pipeline & Coding Workflow Improvements

## Major Work: All-Bash Monitor Scripts with Multi-Iteration Logic

Created comprehensive bash-based monitoring scripts for both coding-workflow and cicd-pipeline skills:

### Architecture
- `monitor.sh` - Orchestration with time-based advancement (MAX_DURATION=120s)
- `common.sh` - Shared utilities (get_state_field, update_state_raw, logging)
- `steps/*.sh` - Individual step scripts with agent fallback
- Agent fallback: if step script missing, delegates to agent via `openclaw send`

### Key Features
- **Multi-iteration advancement**: Reads state → executes step → checks advancement → repeats
- **Reset on progress**: consecutive_no_advancement counter resets when state changes
- **Exit after 2 no-advancement checks**: prevents stuck loops
- **Time budget**: MAX_DURATION=120s safety limit (not MAX_ITERATIONS)
- **Incremental implementation**: Can merge with just 2 example scripts, others use agent fallback

### Example Step Scripts Implemented

**cicd-pipeline/e2e_running.sh:**
- 4-step verification (playwright.config.ts, browserless in config, pnpm test:e2e, container running)
- Any failure → ESCALATED immediately
- Automatic IAP detection (HTTP 200=public, 403/302=IAP enabled)
- Generates IAP token only if needed
- Exports PLAYWRIGHT_BROWSER_WS_ENDPOINT, PLAYWRIGHT_BASE_URL, IAP_ID_TOKEN
- Parses test output for timeouts vs failures
- Routes to DONE/E2E_TIMEOUT_RETRY/E2E_FIXING/ESCALATED

**coding-workflow/testing.sh:**
- Runs unit tests for stack (python/nextjs/monorepo)
- If pass → runs integration tests
- Routes to DONE or FIXING based on results

## Performance Improvement
- Before: BUILDING (10s) → wait 3 min → TESTING (30s) → wait 3 min → DONE = ~6.5 minutes
- After: BUILDING (10s) → TESTING (30s) → DONE in one tick = ~50 seconds
- **Expected reduction: 40-80% on fast paths**

## State Machine Enhancements

### State Classifications
- **Start States**: valid entry points (CODING, BUILDING, FIXING for coding-workflow)
- **Active States**: workflow running, cron monitors
- **Terminal States**: DONE, ESCALATED (workflow ended, cleanup + archive required)

### Terminal State Cleanup (Standardized)
1. Disable monitor cron
2. Kill tmux session (if exists)
3. Archive state file with timestamp + status suffix (-success, -escalated, -failed)
4. Return notification message (cron delivery routes it)

### Cron Frequency
- Updated from 5 min → 3 min (180000ms) for faster feedback

## E2E Test Improvements

### Timeout Handling (NEW)
- Added E2E_TIMEOUT_RETRY state
- Auto-retries timed-out tests with 2x timeout
- Max 2 retry attempts
- Escalates as infrastructure issue if still timing out
- Separate from code fix loop (timeouts ≠ bugs)

### IAP Handling (NEW)
- **Auto-detection**: Checks service without auth first
- HTTP 200 → skip IAP token generation
- HTTP 403/302 → generate IAP token and verify access
- IAP failures → ESCALATED immediately (not fix loop)
- Infrastructure issue, not code bug

### Browserless Enforcement
- E2E tests require Browserless container (ws://localhost:3000)
- Verified before running tests
- No fallback to local Chrome
- Production-safe isolation

## Lessons Learned
- Keep verification simple: 4 clear checks better than complex enforcement patterns
- Auto-detection better than asking: IAP, service names, URLs
- Bash better than Python for command-heavy workflows (no subprocess overhead)
- Agent fallback allows incremental implementation (merge with 2 scripts, implement rest later)
- Time budget (MAX_DURATION) clearer than iteration count
- Separate timeouts from failures (different handling, different root causes)
